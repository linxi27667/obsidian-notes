# 举升机掉电高度丢失与无法下降问题分析

> 2026-05-20 · 问题诊断 · Flash存储 · 掉电保护

---

## 一、问题描述

### 1.1 现象

| 现象 | 描述 |
|------|------|
| 上电后高度为0 | Flash中没有保存历史高度，`pulse_count` 初始化为0 |
| 无法下降 | 举升机在半空中时，按下降键无反应 |
| 高度丢失 | 掉电后再次上电，之前举升的高度数据消失 |

### 1.2 影响

- 举升机举升到半空后掉电，再次上电无法下降
- 必须手动上升触发编码器脉冲，才能恢复下降功能
- 工业场景下存在安全隐患

## 四、解决方案

### 4.1 方案一：双键强制下降（紧急修复）

**目标**：让举升机在高度为0时也能下降

**修改文件**：`APP/Src/key.c`

**修改位置**：`Key_Jog_Start_Check()` 函数开头

**新增代码**：

```c
void Key_Jog_Start_Check(void)
{
    // 【新增】双键同按 = 强制下降（用于上电后高度为0的情况）
    if (g_command.button_up && g_command.button_down 
        && g_command.direction == DIR_STOP) {
        
        // 清除所有限位标志
        g_safety.at_lower_limit = 0;
        g_safety.at_upper_limit = 0;
        g_safety.alarm = ALARM_NONE;
        g_safety.stall_suspected = 0;
        
        // 强制下降
        Motor_Start_All(DIR_DOWN);
        g_command.direction = DIR_DOWN;
        
        #if CTRL_DEBUG == 1
        elog_w("CTRL", "FORCE DOWN (dual button)");
        #endif
        return;
    }
    
    // 原有逻辑保持不变...
}
```

**操作方式**：同时按住上升+下降键 → 强制下降

**优点**：
- 不需要修改Flash相关代码
- 立即可用
- 双键同按是明确的操作意图，安全性可接受

---

### 4.2 方案二：停止时立即保存（推荐）

**目标**：运动停止后立即保存高度，避免掉电丢失

**修改文件**：`APP/Src/app_w25qxx.c`

**修改位置**：`App_W25Qxx_Height_Save_If_Needed()` 函数

**修改代码**：

```c
void App_W25Qxx_Height_Save_If_Needed(void)
{
    static uint32_t last_save_tick = 0;
    static int32_t  last_saved[2]  = {0, 0};

    // 【新增】停止时立即保存
    if (g_command.direction == DIR_STOP) {
        if (g_column[0].pulse_count != last_saved[0]
            || g_column[1].pulse_count != last_saved[1]) {
            App_W25Qxx_Height_Save();
            last_saved[0] = g_column[0].pulse_count;
            last_saved[1] = g_column[1].pulse_count;
        }
        return;
    }

    // 运动中：每1秒保存一次（原来5秒）
    if (g_column[0].pulse_count != last_saved[0]
        || g_column[1].pulse_count != last_saved[1]) {
        uint32_t now = HAL_GetTick();
        if (now - last_save_tick > 1000) {  // 改为1秒
            App_W25Qxx_Height_Save();
            last_saved[0] = g_column[0].pulse_count;
            last_saved[1] = g_column[1].pulse_count;
            last_save_tick = now;
        }
    }
}
```

**保存策略**：

| 场景 | 保存时机 | 丢失量 |
|------|----------|--------|
| 正常停止 | 立即保存 | 0mm |
| 掉电 | 最多丢失1秒数据 | ≤10mm |
| 连续运动中掉电 | 最多丢失1秒数据 | ≤10mm |

**优点**：
- 正常停止时数据完全不丢失
- 掉电丢失量从50mm降低到10mm
- 不需要额外硬件

**效果**：

- 运动中每1秒自动保存一次高度

### 4.4 方案四：掉电检测保护（长期方案）

**目标**：检测24V掉电，在电压跌落瞬间保存高度

**硬件需求**：

| 组件 | 规格 | 作用 |
|------|------|------|
| 电解电容 | 6800μF/35V | 储能，给MCU供电50ms |
| LDO稳压器 | 3.3V | 从24V降压给MCU |
| 电压检测电路 | 分压电阻+比较器 | 检测24V跌落 |
| GPIO中断 | 下降沿触发 | 触发保存函数 |

**电路框图**：

```
24V ──┬── 分压电阻 ── 比较器 ── GPIO中断
      │
      └── 电容 ── LDO ── 3.3V ── MCU
```

**软件实现**：

```c
// 新增：掉电检测中断处理
void HAL_GPIO_EXTI_Callback(uint16_t GPIO_Pin)
{
    if (GPIO_Pin == POWER_FAIL_PIN) {
        // 检测到掉电，立即保存高度
        App_W25Qxx_Height_Save();
        
        // 可选：停止所有电机
        Motor_Stop_All();
    }
}
```

**检测方式选择**：

| 方式 | 响应时间 | 优点 | 缺点 |
|------|----------|------|------|
| ADC轮询 | 1-10ms | 简单 | 响应慢 |
| 比较器+EXTI | 1-10μs | 响应快 | 需要额外硬件 |
| 专用掉电检测IC | <1ms | 最可靠 | 成本高 |

## 六、关键问题解答

### Q1: 为什么上电后高度是0？

**A**: `App_W25Qxx_Height_Load()` 被注释掉了，没有从Flash恢复历史高度。

### Q2: 为什么无法下降？

**A**: `Safety_Check_Lower_Limit()` 检测到 `pulse_count <= 0`，判定已到底部，禁止下降。

### Q3: 双键强制下降安全吗？

**A**: 相对安全。操作者同时按两个键是明确意图，且有机械防撞杆作为最后保护。

### Q4: 停止时保存能避免数据丢失吗？

**A**: 正常停止时能完全避免。只有掉电时会丢失最多1秒的数据（≤10mm）。

### Q5: 没有电容蓄电怎么办？

**A**: 先用停止时保存方案，接受"掉电时最多丢失10mm"。后期再加硬件。

### Q6: Flash写入需要多长时间？

**A**: W25Q128典型写入时间：
- 扇区擦除：45-400ms
- 页编程：0.3-3ms
- 总计：约50-500ms（取决于数据量）

### Q7: 电容需要多大？

**A**: 假设MCU电流50mA，需要50ms完成写入：
- C = I × t / ΔV = 0.05A × 0.05s / 0.5V = 5000μF
- 建议使用6800μF/35V电解电容

### Q8: 为什么先做软件再做硬件？

**A**: 
- 软件先行可以验证逻辑正确性
- 硬件改动不影响软件架构
- 可以并行开发，提高效率
- 降低风险，先跑通功能再优化
