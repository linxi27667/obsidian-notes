# NAT 与 DHCP

## 一、NAT（网络地址转换）

### 1.1 为什么需要 NAT

**类比**：
```
你家只有一条电话线（1 个公网 IP），但家里有 10 个人（10 台设备）都想打电话。

解决办法：你请了一个秘书（路由器/NAT）。
- 家里任何人想打电话，都先告诉秘书要打给谁
- 秘书用那唯一一条电话线打出去，同时记下来是谁在打
- 对方回复时，秘书根据记录转给对应的人

这就是 NAT —— 多个私有 IP 设备共享一个公网 IP。
```

```
问题：家里 10 台设备都只有 1 个公网 IP，怎么同时上网？

解决：NAT 让多个私有 IP 设备共享一个公网 IP。
```

### 1.2 NAT 工作原理

```
┌─────────────────────────────────────────────────┐
│ 你的家庭网络                                     │
│                                                 │
│  手机 192.168.1.10  ──┐                         │
│  电脑 192.168.1.100 ──┤──→ 路由器 192.168.1.1   │
│  ESP32 192.168.1.50──┘       公网IP: 123.45.67.89│
└──────────────────────────┬──────────────────────┘
                           │
                           ▼
                    互联网（公网）
```

**NAT 映射表**（路由器维护，就像秘书的记事本）：

| 内网地址 | 内网端口 | → | 公网 IP | 公网端口 | 目标地址 |
|---------|---------|---|---------|---------|---------|
| 192.168.1.10 | 50000 | → | 123.45.67.89 | 60001 | 8.8.8.8:80 |
| 192.168.1.100 | 50001 | → | 123.45.67.89 | 60002 | 13.212.94.125:1883 |
| 192.168.1.50 | 50002 | → | 123.45.67.89 | 60003 | 13.212.94.125:1883 |

当公网响应回来时，路由器查表，将响应转发给正确的内网设备。

### 1.3 NAT 对 IoT 的影响

**类比**：
```
秘书（NAT 路由器）只会帮你转你主动打出去的电话的回复。
外面的人不能直接打进来找你家某个人 —— 因为秘书不知道要把电话转给谁。

这就是为什么云服务器无法主动连接你家里的 ESP32。
```

```
❌ 云服务器无法主动连接内网 ESP32
   因为 NAT 表中没有对应的映射条目

✅ 解决方案：ESP32 主动连接云服务器
   - ESP32 → MQTT Broker → 保持长连接
   - 云端通过 MQTT 主题下发命令
   - 不需要任何端口映射
```

### 1.4 端口映射（Port Forwarding）

**类比**：
```
如果你确实想让外面的人能打电话给家里某个人，
你可以跟秘书说："凡是打到 8080 号线路的，都转给 192.168.1.100 的 80 号分机。"

这就是端口映射。
```

```
路由器设置：
  外部端口 8080 → 内部 192.168.1.100:80（ESP32 Web Server）

效果：
  外部访问 123.45.67.89:8080 → 路由器转发到 192.168.1.100:80
```

## 二、DHCP（动态主机配置协议）

### 2.1 DHCP 是什么

**类比**：
```
你新搬进一栋大楼（连上 WiFi），你不知道自己的房间号（IP 地址）是多少。

你找大楼管理员（DHCP 服务器/路由器）：
- "请给我分配一个房间号"
- 管理员查看哪些房间空着，给你分配一间
- 告诉你：房间号是 100，这层楼的网关是前台，有问题找前台

这就是 DHCP。
```

### 2.2 DHCP 四步流程（DORA）

```
ESP32 接入 WiFi 后获取 IP 的完整过程：

ESP32                          路由器(DHCP服务器)
  │                                    │
  │◄── WiFi 关联成功                    │
  │                                    │
  │── DHCP Discover (广播) ──────────→│
  │  "有人能给我 IP 吗？"                │
  │                                    │
  │←── DHCP Offer ────────────────────│
  │  "我可以给你 192.168.1.100/24"      │
  │  网关: 192.168.1.1                  │
  │  DNS: 192.168.1.1                  │
  │  租期: 24 小时                      │
  │                                    │
  │── DHCP Request (广播) ───────────→│
  │  "好，我要 192.168.1.100 了！"      │
  │                                    │
  │←── DHCP ACK ──────────────────────│
  │  "确认！IP 归你了"                  │
  │                                    │
  │  配置网络接口:                       │
  │  IP: 192.168.1.100                 │
  │  Mask: 255.255.255.0               │
  │  Gateway: 192.168.1.1              │
  │  DNS: 192.168.1.1                  │
```

### 2.3 ESP-IDF 中 DHCP 的使用

```c
// ESP-IDF 默认启用 DHCP（STA 模式自动获取 IP）
wifi_config_t wifi_config = {
    .sta = {
        .ssid = "MyWiFi",
        .password = "12345678",
    },
};

// 等待 DHCP 分配 IP 的事件
// IP_EVENT_STA_GOT_IP 触发时，IP 已配置完成
```

### 2.4 DHCP vs 静态 IP

| | DHCP（自动） | 静态 IP（手动） |
|------|------|------|
| 配置 | 路由器自动分配 | 手动设置 |
| 适用 | 客户端设备（手机、电脑、ESP32） | 服务器、网关、路由器 |
| 优点 | 即插即用 | 地址固定 |
| 缺点 | 地址可能变化 | 需要手动管理 |
