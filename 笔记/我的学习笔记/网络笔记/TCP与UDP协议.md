# TCP 与 UDP 协议

## 一、TCP（传输控制协议）

```
全称: Transmission Control Protocol
端口: 无固定端口（承载上层协议）
特点: 面向连接、可靠、有序、流量控制
```

### 1.1 核心特性

| 特性 | 说明 |
|------|------|
| 面向连接 | 通信前必须三次握手建立连接 |
| 可靠传输 | 丢包重传、校验和、确认应答（ACK） |
| 有序交付 | 数据包按发送顺序交付给应用层 |
| 流量控制 | 滑动窗口机制，防止发送方压垮接收方 |
| 拥塞控制 | 慢启动、拥塞避免、快速重传、快速恢复 |
| 全双工 | 两个方向可以同时传输数据 |

### 1.2 TCP 报文结构（简化）

```
┌─────────────────────────────────────────┐
│ 源端口(16)  │ 目的端口(16)               │
├─────────────────────────────────────────┤
│ 序列号 (32)                              │
├─────────────────────────────────────────┤
│ 确认号 (32)                              │
├──────┬───┬───┬──────────────────────────┤
│ 首部 │保│U│A│P│R│S│F│   窗口大小(16)     │
│ 长度 │留│R│C│S│S│Y│N│                  │
│      │  │G│K│H│T│N│I│                  │
├─────────────────────────────────────────┤
│      校验和      │    紧急指针           │
├─────────────────────────────────────────┤
│            选项（可选）                   │
├─────────────────────────────────────────┤
│              数据载荷                     │
└─────────────────────────────────────────┘

标志位: URG=紧急 PSH=推送 RST=重置 SYN=同步 FIN=结束
```

### 1.3 适用场景

- MQTT（IoT 设备通信）
- HTTP/HTTPS（Web 服务）
- SSH（远程管理）
- FTP（文件传输）
- 任何不能丢数据的场景

**不适用**：实时音视频（延迟太高）、广播/组播（TCP 只支持点对点）

## 二、UDP（用户数据报协议）

```
全称: User Datagram Protocol
端口: 无固定端口（承载上层协议）
特点: 无连接、不可靠、低延迟、低开销
```

### 2.1 核心特性

| 特性 | 说明 |
|------|------|
| 无连接 | 发送前不需要建立连接 |
| 不可靠 | 不保证到达、不保证顺序、不重传 |
| 低延迟 | 没有握手、没有确认、没有重传等待 |
| 低开销 | 头部仅 8 字节（TCP 最少 20 字节） |
| 支持广播 | 可以向 255.255.255.255 发送 |

### 2.2 UDP 报文结构

```
┌────────────────────────────────┐
│ 源端口(16)  │ 目的端口(16)      │
├────────────────────────────────┤
│ 长度(16)    │ 校验和(16)       │
├────────────────────────────────┤
│         数据载荷                │
└────────────────────────────────┘
总计最少 8 字节头部
```

### 2.3 适用场景

- DNS 查询（53 端口）
- NTP 时间同步（123 端口）
- DHCP（67/68 端口）
- 实时音视频（RTP/RTCP）
- ESP-NOW（WiFi 直连）
- CoAP（IoT 受限设备协议）
- mDNS（5353 端口）

### 2.4 UDP vs TCP 对比

```
TCP: 打电话
  → "喂，听得到吗？" "听得到" "好，我开始了..."
  → 说漏了对方会让你重说
  → 说完对方说"收到"
  → 保证对方完整听到，但有延迟

UDP: 广播/写信
  → 直接说/寄，不管对方听没听到
  → 漏了就漏了，不会重说
  → 最快，但可能丢
```

| 对比项 | TCP | UDP |
|--------|-----|-----|
| 连接 | 面向连接 | 无连接 |
| 可靠性 | 可靠 | 不可靠 |
| 速度 | 慢 | 快 |
| 头部大小 | 最少 20 字节 | 8 字节 |
| 流量控制 | 滑动窗口 | 无 |
| 拥塞控制 | 有 | 无 |
| 广播/组播 | 不支持 | 支持 |

## 三、TCP 三次握手（建立连接）

```
客户端（ESP32）                    服务器（MQTT Broker）
     │                                    │
     │────── SYN (seq=x) ───────────────→│
     │     "我想建立连接"                  │
     │                                    │
     │←──── SYN+ACK (seq=y, ack=x+1) ────│
     │     "好的，我也准备好了"             │
     │                                    │
     │────── ACK (ack=y+1) ─────────────→│
     │     "收到，连接建立！"               │
     │                                    │
     │═══════ 连接已建立，开始传输数据 ═════│
```

### 为什么需要三次？

防止历史重复的连接请求突然到达服务器，导致错误建立连接。

### ESP32 连接 MQTT 时的体现

```
TCP 层: 三次握手 → 建立 192.168.1.100:xxxxx ↔ broker:1883 的 TCP 连接
        ↓
MQTT层: 发送 CONNECT 报文 → 等待 CONNACK
```

## 四、TCP 四次挥手（断开连接）

```
客户端                         服务器
  │                                │
  │── FIN (seq=u) ──────────────→│
  │  "我没有数据要发了"             │
  │                                │
  │←── ACK (ack=u+1) ────────────│
  │  "知道了，但我还有数据要发"      │
  │                                │
  │         (服务器继续发数据...)    │
  │                                │
  │←── FIN+ACK (seq=v, ack=u+1) ──│
  │  "我也发完了"                   │
  │                                │
  │── ACK (ack=v+1) ─────────────→│
  │  "好的，再见！"                 │
  │                                │
  客户端进入 TIME_WAIT (2MSL) → 最终关闭
```

### 为什么需要四次？

因为 TCP 是全双工的，两个方向需要独立关闭。

### TCP 状态速查

| 状态 | 含义 |
|------|------|
| `LISTEN` | 服务器监听端口，等待连接 |
| `SYN_SENT` | 已发送 SYN，等待对方回应 |
| `ESTABLISHED` | 连接已建立，数据传输中 |
| `FIN_WAIT_1` | 己方已发送 FIN，等待确认 |
| `FIN_WAIT_2` | 己方 FIN 已确认，等待对方 FIN |
| `TIME_WAIT` | 等待 2MSL 确保对方收到最后的 ACK |
| `CLOSED` | 连接已完全关闭 |

### 查看 TCP 连接状态

```bash
# Windows
netstat -an | findstr ESTABLISHED

# Linux
netstat -tnp
ss -tnp
```

## 五、TCP 滑动窗口与流量控制

### 5.1 滑动窗口概念

```
TCP 不是发一个包等一个确认，而是可以连续发多个包。

窗口大小 = 发送方在未收到确认前可以发送的最大字节数

┌──────────────────────────────────┐
│    已发送已确认 │ 可发送(窗口) │ 不能发 │
│    ──────────  │  ──────────   │ ───── │
│   1-1000       │  1001-5000    │ 5001+ │
└──────────────────────────────────┘
                ↑
          窗口随着 ACK 滑动前进
```

### 5.2 对 ESP32 的意义

- ESP32 发送 MQTT PUBLISH 报文时，TCP 层自动处理分片和流控
- 发送速度过快 → 窗口满 → 暂停发送 → 等待确认
- 发送速度过慢 → 利用率低 → 适当增大发送缓冲区
