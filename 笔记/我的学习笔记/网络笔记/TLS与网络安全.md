# TLS 与网络安全

## 一、TLS/SSL 是什么

```
TLS 前身: SSL (已弃用 SSLv3)
当前版本: TLS 1.2 / TLS 1.3
特点: 加密传输、身份认证、数据完整性
```

**类比**：
```
没有 TLS 的通信（明文）：
你寄明信片，邮递员、路人都能看到你写了什么。

有 TLS 的通信（加密）：
你把信装进一个只有收信人能打开的保险箱里寄出去。
即使有人截获了保险箱，也打不开看不了内容。
```

### 1.1 明文 vs 加密对比

```
明文 MQTT (1883):
  ESP32 → Broker: {"cmd": "open_door", "pin": 5}
  ↑ 局域网内任何人可抓包看到内容！

加密 MQTT (8883):
  ESP32 → Broker: [加密二进制数据]
  ↑ 即使抓包也无法解析
```

## 二、TLS 握手流程

**类比**：
```
两个人第一次见面想秘密交谈：
1. A: "我支持用中文或英文说暗语"（Client Hello）
2. B: "那我们用中文吧，这是我的身份证"（Server Hello + 证书）
3. A 验证 B 的身份证是真的（验证证书）
4. 两人约定一个只有他们懂的暗号（协商密钥）
5. 开始用暗号交谈（加密通信）
```

**技术流程**：
```
1. Client Hello  →  "我支持 TLS 1.3"
2. Server Hello  →  "用 TLS 1.3，这是我的证书"
3. 验证证书       →  确认证书合法
4. 协商密钥       →  生成会话密钥
5. 加密通信       →  所有数据加密传输
```

## 三、TLS 提供的安全保证

| 保证 | 实现方式 | 说明 |
|------|---------|------|
| 机密性 | 对称加密 (AES-256-GCM) | 即使被截获也无法读取 |
| 完整性 | 消息认证码 (HMAC) | 数据被篡改能检测到 |
| 身份认证 | X.509 证书 + 非对称加密 | 确认对方是真的对方 |
| 前向安全 | ECDHE 密钥交换 | 即使密钥泄露，过去的通信也无法被解密 |

## 四、密码套件

```
TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384
    │     │     │      │          │
    │     │     │      │          └─ 哈希算法（数据完整性校验）
    │     │     │      └─ 对称加密算法（加密数据用的"锁"）
    │     │     └─ 密钥交换算法（两人怎么安全地约定密钥）
    │     └─ 证书类型（身份证是什么材质）
    └─ 协议版本
```

## 五、MQTT + TLS

```
mqtts://broker.emqx.io:8883
     ↑                    ↑
  TLS加密              TLS专用端口
```

- 1883 端口 = MQTT 明文（开发用可以，生产环境不行）
- 8883 端口 = MQTT over TLS（加密，生产环境推荐）

```c
// MQTT over TLS 配置
esp_mqtt_client_config_t mqtt_cfg = {
    .broker.address.uri = "mqtts://broker.emqx.io:8883",
    .broker.verification.certificate = server_cert_pem_start,
};
```

## 六、X.509 证书详解

### 6.1 证书是什么

**类比**：
```
证书就像一个人的"身份证"：
- 身份证上有你的名字、照片、身份证号
- 证书上有"域名"（你是谁）、"公钥"（你的签名）、"过期时间"

身份证由公安局签发，证书由 CA（Certificate Authority，证书颁发机构）签发。

你相信公安局，所以相信身份证。
浏览器相信 CA，所以相信证书。
```

### 6.2 证书包含的内容

```
证书（X.509 v3）包含：
┌──────────────────────────────────┐
│ 版本号                            │
│ 序列号（唯一标识）                  │
│ 签名算法（用什么方式签名的）         │
│ 颁发者（哪个 CA 签发的）            │
│ 有效期（开始时间 ~ 结束时间）        │
│ 主题（这个证书是给谁的）             │
│   - CN = Common Name（域名）       │
│   - O = Organization（组织）       │
│ 公钥（证书持有者的公钥）             │
│ 扩展信息（SAN、用途限制等）          │
│ CA 的数字签名                      │
└──────────────────────────────────┘
```

### 6.3 证书链（信任链）

**类比**：
```
A 是公安局总局（根 CA），不直接给人发身份证。
B 是省公安厅（中间 CA），A 授权 B 可以发。
C 是市公安局（中间 CA），B 授权 C 可以发。
D 是派出所，C 授权 D 给"broker.emqx.io" 发了证书。

验证流程：
D 发的证书 → B 授权了 D → A 授权了 B → A 是可信的根 → 证书可信
```

```
你的浏览器信任 "根 CA"
    │
    └─ 根 CA 签名了 "中间 CA"
           │
           └─ 中间 CA 签名了 "broker.emqx.io 的证书"
                  │
                  └─ 这就是 MQTT 连接时服务器出示的证书

浏览器验证整个链条，全部合法 → 证书可信
```

### 6.4 证书类型

| 类型 | 说明 | 适用场景 |
|------|------|----------|
| DV（域名验证） | 只验证你拥有该域名 | 最常用，签发快 |
| OV（组织验证） | 还要验证组织信息 | 企业网站 |
| EV（扩展验证） | 最严格的验证 | 银行、金融 |
| 通配符证书 | `*.example.com` 匹配所有子域名 | 多个子域名 |
| 自签名证书 | 自己签发，没有 CA 背书 | 开发/内网测试 |

### 6.5 自签名证书 vs CA 证书

```
CA 证书（正规证书）：
  - 由受信任的 CA 签发
  - 浏览器/系统自动信任
  - 生产环境必须用
  - 可能需要付费（Let's Encrypt 免费）

自签名证书（自己签发）：
  - 自己生成，自己信任自己
  - 浏览器/系统不信任，会弹出警告
  - 适合开发环境、内网测试
  - 免费

ESP32 场景：
  - 如果使用 CA 证书 → ESP32 内置信任链，直接验证
  - 如果使用自签名证书 → 需要在 ESP32 代码中指定证书指纹
```

### 6.6 证书的生命周期

```
1. 生成密钥对（公钥 + 私钥）
2. 用公钥生成 CSR（证书签名请求）
3. 提交给 CA
4. CA 验证身份
5. CA 签发证书
6. 部署到服务器
7. 证书过期前续期（Let's Encrypt 90 天）
8. 旧证书吊销/替换
```

### 6.7 ESP32 证书处理

```c
// 方式 1：使用服务器证书验证（推荐）
esp_mqtt_client_config_t mqtt_cfg = {
    .broker.address.uri = "mqtts://broker.emqx.io:8883",
    .broker.verification.certificate = server_cert_pem_start,
};

// 方式 2：使用客户端证书（双向 TLS 认证）
esp_mqtt_client_config_t mqtt_cfg = {
    .broker.address.uri = "mqtts://broker.emqx.io:8883",
    .broker.verification.certificate = server_cert_pem_start,
    .broker.verification.client_certificate = client_cert_pem_start,
    .broker.verification.client_key = client_key_pem_start,
};

// PEM 格式证书示例（存储在 ESP32 NVS 或 Flash 中）
// -----BEGIN CERTIFICATE-----
// MIIDxTCCAq2gAwIBAgIQAf2j6Nrr...
// -----END CERTIFICATE-----
```

## 七、ESP32 安全注意事项

| 风险 | 防护 |
|------|------|
| 明文 MQTT 被抓包 | 使用 TLS（端口 8883） |
| WiFi 密码泄露 | 使用 WPA2/WPA3，NVS 加密存储 |
| MQTT 未授权访问 | 使用用户名密码、ACL |
| OTA 固件被篡改 | 验证固件签名 |
| 证书被窃取 | 使用硬件加密芯片，密钥安全存储 |

## 八、安全协议速查

| 协议 | 层 | 用途 |
|------|-----|------|
| TLS 1.3 | 传输层之上 | 加密 TCP 连接 |
| DTLS 1.3 | UDP 之上 | 加密 UDP 连接（CoAPS） |
| WPA3 | 数据链路层 | WiFi 加密认证 |
| AES-256-GCM | 加密算法 | TLS/ESP-NOW 加密 |
| SHA-256 | 哈希算法 | 完整性校验 |
| RSA-2048 | 非对称加密 | TLS 证书密钥交换 |
| ECDHE | 密钥交换 | TLS 前向安全 |
| X.509 | 证书格式 | TLS 身份认证 |
