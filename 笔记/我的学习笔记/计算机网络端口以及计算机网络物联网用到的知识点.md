# 计算机网络端口 & IoT 网络知识点

## 一、端口基础

### 1.1 什么是端口

端口是操作系统用来区分不同网络服务的**逻辑编号**，范围 0-65535。它与 IP 地址组合成完整的网络地址：`IP:端口`，例如 `192.168.1.100:1883`。

### 1.2 端口分类

| 范围 | 名称 | 说明 |
|------|------|------|
| 0-1023 | **知名端口** | 系统级服务保留，需要 root/管理员权限 |
| 1024-49151 | **注册端口** | 应用程序注册使用，无需特权 |
| 49152-65535 | **动态/私有端口** | 操作系统动态分配给客户端临时使用 |

### 1.3 TCP vs UDP 端口

- **TCP 端口**：面向连接，可靠传输（如 HTTP 80、MQTT 1883）
- **UDP 端口**：无连接，不可靠但快速（如 DNS 53、NTP 123）
- TCP 和 UDP 的端口号是**独立的**，TCP 80 和 UDP 80 是两个不同的端口

## 三、网络模型

### 3.1 OSI 七层模型

```
应用层 (7)    ← HTTP, MQTT, FTP, DNS, SMTP
表示层 (6)    ← 加密、压缩、编码
会话层 (5)    ← 建立、管理、终止会话
传输层 (4)    ← TCP, UDP
网络层 (3)    ← IP, ICMP, ARP
数据链路层 (2) ← 以太网, WiFi, ESP-NOW
物理层 (1)    ← 电缆、光纤、无线电波
```

### 3.2 TCP/IP 四层模型（实际使用）

```
应用层    ← HTTP, MQTT, CoAP, WebSocket
传输层    ← TCP, UDP
网络层    ← IP, ICMP
网络接口层 ← 以太网, WiFi
```

### 3.3 IoT 协议在网络模型中的位置

```
        ┌─────────────────────────────────────┐
应用层   │ MQTT  │ HTTP   │ CoAP  │ WebSocket │
        ├─────────────────────────────────────┤
传输层   │         TCP         │     UDP      │
        ├─────────────────────────────────────┤
网络层   │              IPv4/IPv6              │
        ├─────────────────────────────────────┤
网络接口  │    WiFi    │  以太网  │  BLE  │    │
        └─────────────────────────────────────┘
```

## 四、物联网网络架构

### 4.1 典型 IoT 三层架构

```
┌──────────────────────────────────────────────┐
│              云平台 / 服务器                  │
│    ┌──────────┐  ┌──────────┐  ┌──────────┐  │
│    │ MQTT     │  │ Web      │  │ 数据库   │  │
│    │ Broker   │  │ 服务器   │  │ (MySQL)  │  │
│    └──────────┘  └──────────┘  └──────────┘  │
└──────────────┬───────────────┬───────────────┘
               │ TCP 1883     │ HTTP/WS
               │              │
┌──────────────┼──────────────┼───────────────┐
│    网络层    │  WiFi/4G/5G/以太网              │
└──────────────┼──────────────┼───────────────┘
               │              │
┌──────────────┼──────────────┼───────────────┐
│    感知层    │              │               │
│  ┌──────────┴──┐  ┌────────┴──────┐        │
│  │  ESP32 主控 │  │  ESP32 从设备 │        │
│  │  (网关)     │  │  (传感器节点)  │        │
│  └─────────────┘  └───────────────┘        │
└────────────────────────────────────────────┘
```

### 4.2 IoT 通信模式

| 模式 | 说明 | 典型协议 |
|------|------|----------|
| **发布/订阅** | 设备发布消息到主题，订阅者接收 | MQTT |
| **请求/响应** | 客户端发请求，服务器返回响应 | HTTP, CoAP |
| **推送通知** | 服务器主动推送数据给客户端 | WebSocket |
| **设备到设备** | 设备间直接通信 | ESP-NOW, BLE |

### 4.3 NAT 与内网穿透

```
你的 ESP32 (192.168.1.100)
        │
        ▼
路由器 NAT (192.168.1.1 → 公网 123.45.67.89)
        │
        ▼
互联网 MQTT Broker (broker.emqx.io:1883)
```

- **ESP32 作为客户端**连接外部服务器：没问题，NAT 自动处理
- **外部服务器主动连接 ESP32**：需要端口映射或内网穿透
- **解决方案**：ESP32 主动连接 Broker，通过发布/订阅间接通信

## 五、IP 地址和网络配置

### 5.1 IP 地址分类

| 类型 | 范围 | 说明 |
|------|------|------|
| A 类 | 1.0.0.0 - 126.255.255.255 | 大型网络 |
| B 类 | 128.0.0.0 - 191.255.255.255 | 中型网络 |
| C 类 | 192.0.0.0 - 223.255.255.255 | 小型网络（家用） |

### 5.2 私有 IP 地址（内网）

| 范围 | 常见场景 |
|------|----------|
| 10.0.0.0/8 | 企业内网 |
| 172.16.0.0/12 | 中大型企业 |
| **192.168.0.0/16** | 家庭路由器（最常见） |
| 127.0.0.1 | 本地回环（localhost） |

### 5.3 DHCP vs 静态 IP

| | DHCP（自动） | 静态 IP（手动） |
|------|------|------|
| 配置 | 路由器自动分配 | 手动设置 |
| 适用 | 客户端设备 | 服务器、网关 |
| 优点 | 即插即用 | 地址固定 |
| 缺点 | 地址可能变化 | 需要手动管理 |
| ESP32 使用 | 默认模式 | `esp_netif` 手动配置 |

### 5.4 ESP32 WiFi 连接代码模板

```c
// ESP-IDF WiFi 连接（STA 模式）
wifi_config_t wifi_config = {
    .sta = {
        .ssid = "YourWiFiSSID",
        .password = "YourPassword",
        // 获取 IP 后触发事件
    },
};
esp_wifi_connect();
// 等待 IP_EVENT_STA_GOT_IP 事件
```

## 七、防火墙和端口安全

### 7.1 防火墙规则

```
入站规则（Inbound）：允许外部连接到本机的端口
出站规则（Outbound）：允许本机连接到外部的端口
```

### 7.2 ESP32 安全注意事项

| 风险 | 防护 |
|------|------|
| 明文 MQTT 被抓包 | 使用 TLS（端口 8883） |
| WiFi 密码泄露 | 使用 WPA2/WPA3，NVS 加密存储 |
| MQTT 未授权访问 | 使用用户名密码、ACL |
| OTA 固件被篡改 | 验证固件签名 |

## 九、IoT 常用协议对比

| 协议 | 传输层 | 开销 | QoS | 适用场景 |
|------|--------|------|-----|----------|
| **MQTT** | TCP | 最小 2 字节 | 3 级 | IoT 设备通信（推荐） |
| **HTTP** | TCP | 大（Headers 多） | 无 | Web API、REST |
| **CoAP** | UDP | 小 | 4 级 | 低功耗受限设备 |
| **WebSocket** | TCP | 中等 | 无 | 浏览器实时通信 |
| **ESP-NOW** | WiFi | 极小 | 无 | ESP32 设备直连 |
| **BLE GATT** | BLE | 小 | 无 | 短距离低功耗 |

## 十、WebSocket 与 HTTP 对比

### 10.1 HTTP 请求/响应模式

```
客户端                    服务器
  │── GET /api/status ──→│
  │←── 200 OK + JSON ───│
  │（连接关闭）            │
```

- 每次请求建立新连接（HTTP/1.1）
- 服务器不能主动推送数据
- 适合：获取状态、发送命令

### 10.2 WebSocket 全双工模式

```
客户端                    服务器
  │── HTTP 升级请求 ────→│
  │←── 101 Switching ───│
  │←═════════════════════║ 双向数据传输
  │  ←─ 服务器推送 ──────║
  │──→ 客户端发送 ───────║
  │←═════════════════════║
```

- 一次握手，持久连接
- 双向实时通信
- 适合：AI 对话、实时数据流

### 10.3 MQTT over WebSocket

MQTT 可以通过 WebSocket 传输（端口 9001），适用于：
- 浏览器中直接使用 MQTT（Web MQTT 客户端）
- 穿越严格防火墙（只允许 80/443 端口）
- 小智项目中的 WebSocket 协议就用于 AI 对话

## 十一、TLS/SSL 加密

### 11.1 为什么需要加密

```
明文 MQTT (1883):
  ESP32 → Broker: {"cmd": "open_door", "pin": 5}
  ↑ 局域网内任何人可抓包看到内容！

加密 MQTT (8883):
  ESP32 → Broker: [加密二进制数据]
  ↑ 即使抓包也无法解析
```

### 11.2 TLS 握手流程

```
1. Client Hello  →  "我支持 TLS 1.3"
2. Server Hello  →  "用 TLS 1.3，这是我的证书"
3. 验证证书       →  确认证书合法
4. 协商密钥       →  生成会话密钥
5. 加密通信       →  所有数据加密传输
```

### 11.3 ESP32 使用 TLS

```c
// MQTT over TLS 配置
esp_mqtt_client_config_t mqtt_cfg = {
    .broker.address.uri = "mqtts://broker.emqx.io:8883",
    .broker.verification.certificate = server_cert_pem_start,
};
```

## 十二、实战：ESP32 MQTT 网络流程

```
┌─ ESP32 启动 ─────────────────────────────────┐
│                                               │
│  1. WiFi 初始化                                │
│     esp_wifi_init() → esp_wifi_start()        │
│                                               │
│  2. 连接 WiFi AP                               │
│     esp_wifi_connect()                        │
│     ↓ 等待 IP_EVENT_STA_GOT_IP 事件            │
│     获取 IP: 192.168.1.100                     │
│                                               │
│  3. DNS 解析                                   │
│     broker.emqx.io → 13.212.94.125            │
│                                               │
│  4. TCP 连接到 Broker                          │
│     connect(13.212.94.125, 1883)              │
│     ↓ TCP 三次握手                             │
│                                               │
│  5. MQTT CONNECT                               │
│     发送 MQTT CONNECT 报文                     │
│     Client ID: "xiaozhi_master"               │
│     ↓ 收到 CONNACK (返回码 0 = 成功)           │
│                                               │
│  6. 订阅主题                                   │
│     SUBSCRIBE → "xiaozhi/iot/cmd/#"            │
│     ↓ 收到 SUBACK                              │
│                                               │
│  7. 发布上线通知                               │
│     PUBLISH → "xiaozhi/iot/announce"           │
│     消息: {"device":"xiaozhi_master","online":true} │
│                                               │
│  8. 进入事件循环                               │
│     等待消息、发送心跳、处理命令...              │
│                                               │
│  9. 心跳保活 (Keepalive = 240s)                │
│     每 90s 发送 PINGREQ → 收到 PINGRESP        │
│                                               │
│ 10. 断线重连 (60s 间隔)                        │
│     连接断开 → 等待 60s → 重新连接              │
└───────────────────────────────────────────────┘
```

## 十三、常见问题速查

### 端口被占用

```bash
# Windows 查看占用
netstat -ano | findstr :1883
# 用 PID 到任务管理器找进程

# 解决
taskkill /PID <PID> /F
```

### ESP32 连接不上 Broker

| 排查步骤 | 命令/操作 |
|----------|----------|
| 1. WiFi 是否连上 | 串口日志看 `got ip` |
| 2. Broker 是否在运行 | `docker ps` |
| 3. 端口是否开放 | `nc -zv 192.168.1.x 1883` |
| 4. 防火墙是否放行 | Windows Defender 防火墙 |
| 5. Broker 日志 | `docker logs mosquitto` |

### ESP32 在同一网络但连不上

- 确认 ESP32 和电脑在**同一个子网**（如都在 192.168.1.x）
- ESP32 连接的是电脑所在 WiFi，不是手机热点
- 路由器没有开启 AP 隔离
